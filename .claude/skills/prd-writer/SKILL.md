Always use this skill for any feature definition task, even if the user doesn't say "PRD" explicitly. --- # PRD writer ## What this skill does Produces a structured, actionable PRD from a feature description. The output is the single source of truth for implementation and QA. ## Preconditions — ask if missing Before writing, confirm you have: - Product/app name and one-line description - User persona (who is affected by this feature) - Problem statement (what pain does this solve) - Rough scope signal (small/medium/large, or time constraint) If any are missing, ask for them before producing the PRD. Do not guess or invent personas. ## Required sections (always include all seven) ### 1. Problem State the user pain in one paragraph. No solutions here. Bad: "Users can't export data, so we need a CSV button." Good: "Users lose context when switching between tools because there is no way to move their data out of the product." ### 2. Goal One sentence. What does success look like for the user? Not "we will build X" — "users will be able to Y." ### 3. User stories 3–5 stories. Format: "As a [persona], I want [action] so that [outcome]." Cover the happy path and the most important edge case. ### 4. Acceptance criteria Numbered list. Each item must be binary (pass or fail, not "works well"). Bad: "Export should be fast." Good: "CSV export of 1,000 rows completes in under 3 seconds." Minimum 5 criteria. These become the QA checklist. ### 5. Out of scope List 3–5 things this version explicitly does NOT include. This prevents scope creep in implementation and review. ### 6. Open questions Things that need a decision before or during build. Flag the owner and priority (blocking / non-blocking). ### 7. Success metrics How will you know in 2 weeks if this shipped well? One primary metric (quantitative). One secondary signal. ## Judgment rules - If the problem statement contains a solution, rewrite it as pure pain. - If acceptance criteria use "should", rewrite as "must" or "does not". - If scope is unclear, write the MVP version and note what was cut. - If the user gives a vague request, write a minimal PRD and ask one clarifying question at the end, not five. - Never pad sections. Short and specific beats long and vague. ## Voice Direct. No "robust", "leverage", "seamless", "delve", "comprehensive." Write like a PM briefing an engineer who will start coding in an hour. ## Handoff After producing the PRD, end with: "Run /plan-eng-review to check feasibility, then /autoplan to generate an implementation plan."
